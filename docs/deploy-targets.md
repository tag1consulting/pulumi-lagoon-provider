# Managing Lagoon Deploy Targets with Pulumi

This guide covers how to manage Lagoon deploy targets and deploy target configurations using the Pulumi Lagoon provider.

## Overview

Deploy targets represent Kubernetes clusters registered with Lagoon as deployment destinations. Projects can be routed to different deploy targets based on branch patterns, enabling multi-cluster architectures (e.g., production and non-production clusters).

**Key Concepts:**
- **DeployTarget** - A Kubernetes cluster registered with Lagoon
- **DeployTargetConfig** - Branch pattern routing rules that determine which deploy target receives deployments

## DeployTarget Resource

Represents a Kubernetes cluster registered as a deployment target in Lagoon.

### Example

```python
from pulumi_lagoon import DeployTarget, DeployTargetArgs

prod_cluster = DeployTarget("production",
    DeployTargetArgs(
        name="production-cluster",
        kubernetes_endpoint="https://k8s-prod.example.com:6443",
        kubernetes_token="<cluster-token>",
        console_url="https://console-prod.example.com",
        ssh_host="ssh.lagoon-prod.example.com",
        ssh_port=32222,
        router_pattern="${project}.${environment}.prod.example.com",
    )
)
```

### Properties

| Name | Type | Required | Description |
|------|------|----------|-------------|
| `name` | `str` | Yes | Deploy target name (unique identifier) |
| `kubernetes_endpoint` | `str` | Yes | Kubernetes API endpoint URL |
| `kubernetes_token` | `str` | Yes | Service account token for cluster access |
| `console_url` | `str` | No | URL to Kubernetes dashboard/console |
| `ssh_host` | `str` | No | SSH hostname for direct pod access |
| `ssh_port` | `int` | No | SSH port (default: 32222) |
| `router_pattern` | `str` | No | Default route pattern for autogenerated routes |

### Router Pattern

The `router_pattern` defines how autogenerated routes are constructed. It supports template variables:

- `${project}` - Project name
- `${environment}` - Environment name
- `${service}` - Service name (optional)

**Examples:**
- `${project}.${environment}.example.com` → `mysite.main.example.com`
- `${environment}.${project}.k8s.example.com` → `main.mysite.k8s.example.com`
- `${project}-${environment}.prod.example.com` → `mysite-main.prod.example.com`

### Outputs

- `lagoon_id` - The Lagoon internal deploy target ID
- `name` - Deploy target name
- All other input properties

### Resource ID

Numeric Lagoon deploy target ID (e.g., `1`)

## DeployTargetConfig Resource

Routes deployments to specific deploy targets based on branch patterns. Multiple configurations can exist for a project, evaluated in order until a match is found.

### Example

```python
from pulumi_lagoon import DeployTargetConfig, DeployTargetConfigArgs

prod_config = DeployTargetConfig("prod-branches",
    DeployTargetConfigArgs(
        project_id=123,
        deploytarget_id=1,
        branches="^(main|production)$",
        weight=1,
    )
)

nonprod_config = DeployTargetConfig("nonprod-branches",
    DeployTargetConfigArgs(
        project_id=123,
        deploytarget_id=2,
        branches="^(develop|stage|feature/.*)$",
        weight=2,
    )
)
```

### Properties

| Name | Type | Required | Description |
|------|------|----------|-------------|
| `project_id` | `int` | Yes | Project ID to apply routing to |
| `deploytarget_id` | `int` | Yes | Deploy target ID to route matching branches to |
| `branches` | `str` | No | Regular expression for branch matching |
| `pull_requests` | `str` | No | Regular expression for PR matching |
| `weight` | `int` | No | Priority (lower = higher priority, default: 0) |

### Branch Pattern Matching

The `branches` field accepts regular expressions that match against Git branch names:

**Common Patterns:**
- `^main$` - Exact match: main branch only
- `^(main|production)$` - Multiple exact matches
- `^(main|develop|stage)$` - Production and staging branches
- `^feature/.*` - All feature branches
- `.*` - Match all branches (catch-all)

### Pull Request Pattern Matching

The `pull_requests` field works the same as `branches` but matches PR titles or branch names:

**Examples:**
- `^hotfix/.*` - PRs from hotfix branches
- `.*` - All pull requests

### Weight and Evaluation Order

When multiple configurations exist for a project:
1. Configurations are sorted by weight (ascending)
2. Each pattern is evaluated in order
3. The first matching pattern determines the deploy target
4. If no patterns match, the project's default `deploytarget_id` is used

**Weight Examples:**
- `weight=1` - Highest priority (evaluated first)
- `weight=2` - Lower priority
- `weight=0` - Default (same as weight=1)

### Outputs

- `lagoon_id` - The Lagoon internal config ID
- `project_id` - Project ID
- `deploytarget_id` - Deploy target ID
- `branches` - Branch pattern
- `pull_requests` - PR pattern
- `weight` - Weight value

### Resource ID

`{project_id}:{config_id}` (e.g., `123:5`)

## Multi-Cluster Routing Example

Here's a complete example showing production/non-production cluster separation:

```python
import pulumi
from pulumi_lagoon import (
    DeployTarget, DeployTargetArgs,
    DeployTargetConfig, DeployTargetConfigArgs,
    Project, ProjectArgs,
)

# === Deploy Targets (Clusters) ===

# Production cluster
prod_cluster = DeployTarget("production",
    DeployTargetArgs(
        name="production-cluster",
        kubernetes_endpoint="https://k8s-prod.example.com:6443",
        kubernetes_token="<prod-cluster-token>",
        console_url="https://console-prod.example.com",
        ssh_host="ssh.lagoon-prod.example.com",
        ssh_port=32222,
        router_pattern="${project}.${environment}.prod.example.com",
    )
)

# Non-production cluster
nonprod_cluster = DeployTarget("nonprod",
    DeployTargetArgs(
        name="nonprod-cluster",
        kubernetes_endpoint="https://k8s-nonprod.example.com:6443",
        kubernetes_token="<nonprod-cluster-token>",
        console_url="https://console-nonprod.example.com",
        ssh_host="ssh.lagoon-nonprod.example.com",
        ssh_port=32222,
        router_pattern="${project}.${environment}.nonprod.example.com",
    )
)

# === Project ===

# Default to production cluster
project = Project("drupal-site",
    ProjectArgs(
        name="drupal-site",
        git_url="git@github.com:example/drupal-site.git",
        deploytarget_id=prod_cluster.lagoon_id,  # Default
        production_environment="main",
        branches="^(main|develop|stage|feature/.*)$",
        pullrequests=".*",
    )
)

# === Deploy Target Configs (Routing Rules) ===

# Route production branches to production cluster
prod_routing = DeployTargetConfig("prod-routing",
    DeployTargetConfigArgs(
        project_id=project.lagoon_id,
        deploytarget_id=prod_cluster.lagoon_id,
        branches="^(main|production)$",
        weight=1,  # Highest priority
    ),
    opts=pulumi.ResourceOptions(depends_on=[project, prod_cluster])
)

# Route staging branches to non-production cluster
stage_routing = DeployTargetConfig("stage-routing",
    DeployTargetConfigArgs(
        project_id=project.lagoon_id,
        deploytarget_id=nonprod_cluster.lagoon_id,
        branches="^(develop|stage)$",
        weight=2,
    ),
    opts=pulumi.ResourceOptions(depends_on=[project, nonprod_cluster])
)

# Route feature branches to non-production cluster
feature_routing = DeployTargetConfig("feature-routing",
    DeployTargetConfigArgs(
        project_id=project.lagoon_id,
        deploytarget_id=nonprod_cluster.lagoon_id,
        branches="^feature/.*",
        weight=3,
    ),
    opts=pulumi.ResourceOptions(depends_on=[project, nonprod_cluster])
)

# Route all PRs to non-production cluster
pr_routing = DeployTargetConfig("pr-routing",
    DeployTargetConfigArgs(
        project_id=project.lagoon_id,
        deploytarget_id=nonprod_cluster.lagoon_id,
        pull_requests=".*",
        weight=4,
    ),
    opts=pulumi.ResourceOptions(depends_on=[project, nonprod_cluster])
)

# Exports
pulumi.export("prod_cluster_id", prod_cluster.lagoon_id)
pulumi.export("nonprod_cluster_id", nonprod_cluster.lagoon_id)
pulumi.export("project_id", project.lagoon_id)
```

**Routing Behavior:**
1. `main` or `production` branches → Production cluster (weight 1)
2. `develop` or `stage` branches → Non-production cluster (weight 2)
3. `feature/*` branches → Non-production cluster (weight 3)
4. Pull requests → Non-production cluster (weight 4)
5. Any other branches → Production cluster (project default)

## Geographic Region Routing

You can use deploy target configs to route projects by region:

```python
from pulumi_lagoon import DeployTarget, DeployTargetArgs, DeployTargetConfig, DeployTargetConfigArgs

# US cluster
us_cluster = DeployTarget("us-east",
    DeployTargetArgs(
        name="us-east-cluster",
        kubernetes_endpoint="https://k8s-us.example.com:6443",
        kubernetes_token="<us-token>",
        router_pattern="${project}.${environment}.us.example.com",
    )
)

# EU cluster
eu_cluster = DeployTarget("eu-west",
    DeployTargetArgs(
        name="eu-west-cluster",
        kubernetes_endpoint="https://k8s-eu.example.com:6443",
        kubernetes_token="<eu-token>",
        router_pattern="${project}.${environment}.eu.example.com",
    )
)

# US-based project
us_project = Project("us-site",
    ProjectArgs(
        name="us-site",
        git_url="git@github.com:example/us-site.git",
        deploytarget_id=us_cluster.lagoon_id,
        branches="^(main|develop)$",
    )
)

# EU-based project
eu_project = Project("eu-site",
    ProjectArgs(
        name="eu-site",
        git_url="git@github.com:example/eu-site.git",
        deploytarget_id=eu_cluster.lagoon_id,
        branches="^(main|develop)$",
    )
)
```

## GraphQL API Reference

The provider uses these Lagoon GraphQL operations under the hood.

### DeployTarget Operations

**Create a deploy target (Kubernetes cluster):**

```graphql
mutation AddKubernetes($input: AddKubernetesInput!) {
    addKubernetes(input: $input) {
        id
        name
        consoleUrl
        sshHost
        sshPort
        routerPattern
        created
    }
}
```

**Query all deploy targets:**

```graphql
query AllKubernetes {
    allKubernetes {
        id
        name
        consoleUrl
        sshHost
        sshPort
        routerPattern
    }
}
```

### DeployTargetConfig Operations

**Create a deploy target config (routing rule):**

```graphql
mutation AddDeployTargetConfig($input: AddDeployTargetConfigInput!) {
    addDeployTargetConfig(input: $input) {
        id
        weight
        branches
        pullrequests
        deployTarget { id name }
        project { id name }
    }
}
```

**Query configs by project ID:**

```graphql
query DeployTargetConfigsByProjectId($project: Int!) {
    deployTargetConfigsByProjectId(project: $project) {
        id
        weight
        branches
        pullrequests
        deployTarget { id name }
        project { id name }
    }
}
```

> **Import tip:** Use the query above to find config IDs. The import format is `{project_id}:{config_id}` (e.g., `123:5`).

## Importing Existing Deploy Targets

### Import ID Formats

| Resource | Format | Example |
|----------|--------|---------|
| `DeployTarget` | `{numeric_id}` | `1` |
| `DeployTargetConfig` | `{project_id}:{config_id}` | `123:5` |

### Import Examples

```bash
# Import a deploy target
pulumi import lagoon:lagoon:DeployTarget prod-cluster 1

# Import a deploy target config
pulumi import lagoon:lagoon:DeployTargetConfig prod-routing 123:5
```

After importing, add the corresponding resource definition to your Pulumi code:

```python
from pulumi_lagoon import DeployTarget, DeployTargetArgs

# After importing deploy target ID 1
prod_cluster = DeployTarget("production",
    DeployTargetArgs(
        name="production-cluster",
        kubernetes_endpoint="https://k8s-prod.example.com:6443",
        kubernetes_token="<cluster-token>",
        router_pattern="${project}.${environment}.prod.example.com",
    ),
    opts=pulumi.ResourceOptions(import_="1")
)
```

To find deploy target IDs and config IDs, use the Lagoon CLI:

```bash
# List deploy targets
lagoon list deploy-targets

# Show project deploy target configs
lagoon get project --project my-project
```

## Validation Rules

### Deploy Target Names
- Must be unique across all deploy targets
- No special validation rules

### Kubernetes Endpoint
- Must be a valid HTTPS URL
- Should be reachable from the Lagoon core installation

### Branch and PR Patterns
- Must be valid regular expressions
- Empty string matches nothing
- `.*` matches everything (catch-all)

### Weight
- Must be a non-negative integer
- Lower values = higher priority
- Default: 0

## Troubleshooting

### Deployments Not Routing Correctly

1. **Check configuration order** - List configs by weight: lower weight = higher priority
2. **Test regex pattern** - Verify branch name matches pattern (case-sensitive)
3. **Check default** - If no patterns match, the project's default `deploytarget_id` is used
4. **Review Lagoon logs** - Check `lagoon-core` logs for routing decisions

### Cluster Connection Fails

1. **Verify endpoint** - Ensure the Kubernetes API is reachable from Lagoon core
2. **Check token** - Verify the service account token has required permissions
3. **Test manually** - Use `kubectl --server=<endpoint> --token=<token> get namespaces`

### SSH Access Not Working

1. **Check SSH host** - Verify `ssh_host` is reachable from client
2. **Verify port** - Default is 32222; check firewall rules
3. **Test connection** - Use `ssh -p 32222 <ssh_host>` to verify connectivity

### Import Fails

1. **Deploy target not found** - Verify it exists: `lagoon list deploy-targets`
2. **Config not found** - Check project details: `lagoon get project --project <name>`
3. **Wrong format** - Use numeric IDs, not names

### Router Pattern Issues

1. **Routes not generating** - Check that `router_pattern` is set and valid
2. **Wrong format** - Use `${project}`, `${environment}`, `${service}` placeholders
3. **DNS not resolving** - Ensure wildcard DNS points to cluster ingress

## Best Practices

### Deploy Target Naming
- Use descriptive names that indicate purpose: `production-cluster`, `nonprod-cluster`
- Include region if deploying multi-region: `us-east-prod`, `eu-west-nonprod`

### Branch Pattern Strategy
- **Start specific, end general** - Use low weights for specific branches (main, production), higher weights for catch-all patterns
- **Test patterns** - Use a regex tester to verify patterns match expected branches
- **Document routing** - Comment complex routing rules in your code

### Security
- **Rotate tokens** - Kubernetes tokens should be rotated regularly
- **Least privilege** - Service account should have minimum required permissions
- **Separate clusters** - Use distinct clusters for prod/nonprod for security isolation

### Router Patterns
- **Keep it simple** - `${project}.${environment}.domain.com` works for most cases
- **Consider DNS** - Ensure your DNS can handle the pattern (wildcard records)
- **Test in non-prod** - Verify route generation before deploying to production

### Multi-Cluster Architecture
- **Separate by environment** - Production on one cluster, non-production on another
- **Consider cost** - Non-production can use smaller/cheaper clusters
- **Plan networking** - Ensure clusters can reach shared services (databases, caches)
